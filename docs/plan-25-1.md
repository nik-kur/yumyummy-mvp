# План доработок 25(1)

Рабочий план для следующей версии приложения (TestFlight-билд поверх 25(0),
которая сейчас на проверке в App Store). Файл — единый источник правды для
чатов с агентом: в новом чате достаточно сослаться на этот файл и назвать
номер пачки.

## Контекст проекта

- **Продукт**: YumYummy — трекер питания (iOS, Expo/React Native + FastAPI на
  Render + Neon Postgres). Логирование еды текстом/фото/голосом, ИИ-агент
  считает КБЖУ и даёт ссылку на источник.
- **Агент v2** (`app/agent_v2/`) задеплоен в прод: единый вызов LLM + USDA FDC
  + проверка живости ссылок. Включается по фиче-флагу
  `AGENT_V2_ACCOUNT_IDS` в `app/api/app_api.py`; все остальные юзеры (включая
  ревьюеров Apple) — на v1. При ошибке v2 автоматический фолбэк на v1.
- Агент уже умеет: intent-роутинг (auto), фото с этикеткой → цифры с
  этикетки без поиска (`nutrition_source`: label / label_partial / estimate),
  брендовые запросы → официальный сайт или магазин/агрегатор (мертвые ссылки
  отфильтровываются), безбрендовые блюда → разбивка на компоненты со ссылками
  на USDA (fdc.nal.usda.gov), штрихкоды → OpenFoodFacts.

## Жёсткие правила для всех пачек

1. **Не ломать 25(0)**: серверный API меняем только аддитивно (новые поля,
   новые эндпоинты). Контракт `WorkflowRunResponse` (intent, message_text,
   confidence, totals, items[], source_url) должен продолжать работать как есть.
2. Мобильные изменения едут в новый TestFlight-билд (поднять buildNumber в
   EAS), App Store-версия 25(0) не трогается.
3. После серверных изменений — деплой на Render (`yumyummy-mvp-eu`,
   автодеплой по пушу в main) и проверка `GET /health`.
4. Для UI-ресёрча доступен Mobbin MCP (`user-Mobbin`).

## Пачка 1 — быстрые UI-фиксы (задачи 3, 4, 1)

**Задача 3. Виджет: выровнять layout.** Наполнение и идея ок, но алаймент
кривой (кольцо %, цифры kcal left, строки P/F/C, сетка кнопок Text/Photo/
Voice/Saved). Привести к аккуратной сетке в обоих размерах виджета
(small + medium). Скрины «как сейчас» — у Никиты.

**Задача 4. Экран логирования (capture.tsx): вылизать layout.** Известные
проблемы: гэп между клавиатурой и кнопками/полем ввода; невыровненный шрифт
в поле ввода. Функциональность вся на месте — менять только расположение/
отступы/типографику. Перед правками посмотреть референсы на Mobbin
(Cal AI, MyFitnessPal, Yazio — экраны quick log).

**Задача 1. Красивое ожидание поиска.** Агент v2 отвечает ~8–12 с на
брендовые запросы (текст без бренда ~3–5 с, фото ~3–15 с). Вместо простого
спиннера: обратный отсчёт/прогресс на ~10 с; если ответ не пришёл за 10 с —
мягкое сообщение «ещё секунду…». Продумать статусные подписи (что агент
сейчас делает) — можно фейковые стадии, можно честные.

## Пачка 2 — редактирование блюд (задача 2)

Самая большая задача. Возможности:

- Редактировать блюдо и в логе дня, и в сохранённом меню.
- Два режима правки: граммовка (с автопересчётом КБЖУ пропорционально) ИЛИ
  прямое редактирование КБЖУ.
- Правки на уровне каждого компонента блюда (items[]), не только тоталов.
- Удаление отдельного компонента (пример: салат по фото разбит на компоненты,
  помидоры не ел — удалил, тотал пересчитался).
- В сохранённом меню — редактирование названия блюда.

Технически: меals хранятся на сервере (таблица meals + meal_items, alembic-
миграция `add_meal_items_breakdown` уже есть). Нужны PATCH-эндпоинты
(аддитивно!), пересчёт тоталов на сервере, оптимистичный UI на мобилке.
UX-паттерн: bottom sheet со степпером граммов / полями КБЖУ.

## Пачка 3 — агент: прозрачность оценки и мульти-фото (задачи 7, 8)

**Задача 7. Exact/estimate: пересмотреть правила и объяснять оценку.**
Сейчас confidence HIGH/ESTIMATE местами неинтуитивен (например, этикетка с
полными КБЖУ на фото должна быть exact). Агент v2 внутри уже знает метод:
этикетка с фото / официальный сайт / магазин / агрегатор / USDA-компоненты /
чистая оценка. Сделать:
- Ревизия правил confidence в `app/agent_v2/pipelines/*` (label → HIGH уже
  есть; перепроверить пограничные случаи).
- Новое аддитивное поле в ответе API (например `assessment`: машиночитаемый
  метод + короткий человеческий текст «КБЖУ прочитаны с этикетки на фото,
  порцию оценил по виду» / «нашёл точные данные на официальном сайте бренда»).
- На карточке блюда (голубая плашка с источниками) показывать этот текст
  ПЕРЕД ссылками. Тексты локализуются (ru/en) на мобилке по коду метода.

**Задача 8. Несколько фото в одном инпуте.** Мобилка: мультивыбор в пикере
(и несколько снимков подряд с камеры), превью-чипсы на capture-экране.
Сервер: `workflow/run` принимает массив изображений (аддитивно к текущему
одиночному полю). Агент: фото-пайплайн запускается на каждое фото
параллельно (`asyncio.gather`), результаты мёржатся в один ответ (items
конкатенируются, тоталы суммируются). Ограничение: 3–5 фото на запрос.

## Пачка 4 — аналитика (задачи 5, 6)

**Задача 5. Week-вкладка: история глубже + интереснее.**
- Пагинация назад по неделям (свайп/стрелки), не только текущая неделя.
- Ресёрч на Mobbin + свои идеи для новых блоков: серия дней в цели (streak),
  средние КБЖУ по неделям, распределение белка по дням, тепловая карта
  месяца (уже есть прототип?), тренд веса vs калории, «лучший/худший день»,
  разбивка по приёмам пищи и т.п. Сначала концепт-скетч, согласовать, потом
  строить.

**Задача 6. Week in Recap (супер-фича).** По воскресеньям пуш с короткой
классной аналитикой недели. Компоненты:
- Серверная агрегация недели (meals + weight check-ins) + LLM-генерация
  короткого дружелюбного текста-резюме (можно через дешёвый flash-вызов).
- Пуш-инфраструктура: expo-notifications + серверная рассылка по расписанию
  (Render cron job). Проверить, что уже есть в `notifications.tsx`.
- Экран/шторка Recap в приложении (красивые карточки, можно шарить).
- Ресёрч на Mobbin: Spotify Wrapped-паттерны, Strava/Whoop weekly recap.
